# Agent Doctrine

Этот документ фиксирует направление проекта. Это не README и не список
команд. Это принципы, по которым агент должен развиваться из текущего MVP в
универсальную автономную систему.

## Главный Тезис

Проект строит не "агента-фрилансера" и не "чат с tools".

Проект строит личную операционную систему автономного агента, которая со
временем должна уметь:

- понимать человеческие задачи;
- выбирать инструменты;
- проверять источники;
- учиться на доказательствах;
- чинить себя под контролем;
- улучшать свою архитектуру;
- создавать под-агентов;
- работать по расписанию;
- отчитываться человеку понятным языком.

Но агент не должен становиться хаотичной системой, которая делает что хочет.
Любая автономность проходит через ограничения, логи, тесты, approval,
rollback, evidence, source ranking и budget limits.

## Что Это Не Такое

Агент не является одной LLM.

Агент не является свободным процессом, который сам себе всё разрешает.

Агент не должен сохранять всё подряд в память.

Агент не должен молча менять код, политики, внешние каналы или тратить деньги.

Агент не должен говорить пользователю техническим мусором, когда нужен
человеческий ответ.

## Базовая Архитектурная Истина

LLM является модулем рассуждения, а не хозяином системы.

Правильная модель:

```text
Human
  -> Agent Core
      -> Planner / Reasoning Model
      -> Tools
      -> Memory
      -> Evidence
      -> Policy
      -> Budget
      -> Approval
      -> Runtime
```

LLM может предлагать план, объяснение, patch, текст или гипотезу.

Agent Core обязан проверять:

- источники;
- тесты;
- логи;
- политики;
- безопасность;
- бюджет;
- approval;
- возможность rollback.

## Путь Автономии

Текущий проект является ядром будущего агента. Он уже умеет выполнять
контролируемые циклы, но полноценная автономность начинается только после
появления runtime-слоя.

Ориентировочная шкала:

- Level 0: ручной чат без tools.
- Level 1: агент отвечает и иногда использует tools.
- Level 2: агент планирует, действует, проверяет, пишет логи и просит approval.
- Level 3: агент сам запускает bounded dry-run циклы, ведёт task queue и
  создаёт отчёты.
- Level 4: агент может выполнять ограниченные изменения после policy checks,
  tests и approval.
- Level 5: агент работает почти самостоятельно в заданном operational domain,
  с budget governor, circuit breaker, audit, rollback и human oversight.

Цель проекта не в том, чтобы сразу дать Level 5. Цель в том, чтобы каждый
следующий уровень был проверяемым и управляемым.

## Обучение

Агент должен учиться не поглощением, а через доказательства.

Правильный поток:

```text
source
  -> claim
  -> verification
  -> source ranking
  -> conflict check
  -> confidence
  -> memory write policy
  -> provenance-backed memory
```

Каждое знание должно иметь:

- откуда оно взято;
- когда прочитано;
- какой claim извлечён;
- чем claim подтверждён;
- есть ли конфликт;
- какой confidence;
- можно ли использовать это знание в текущей роли.

Агент не должен использовать память как попугай. Knowledge Use Policy должен
решать, когда знание уместно, а когда оно не относится к задаче.

## Саморемонт

Саморемонт не означает "агент сам себя переписал".

Правильный поток:

```text
logs / failed tests / code
  -> diagnose
  -> repair proposal
  -> diff
  -> approval
  -> apply
  -> run tests
  -> success or rollback
  -> audit
```

Запрещённый поток:

```text
agent guessed
  -> wrote code
  -> hoped it works
```

Саморемонт должен быть доказуемым: причина, целевой файл, diff, тесты,
confidence и rollback path.

## Саморазвитие

Универсальный агент не обязан уметь всё сразу.

Универсальный агент должен уметь расширять свои способности:

```text
limitation detected
  -> learning task
  -> design proposal
  -> patch proposal
  -> tests
  -> approval
  -> implementation
  -> audit
  -> memory update
```

Новые инструменты, роли, политики, под-агенты и runtime-режимы добавляются
только через governance.

## Человеческий Интерфейс

Внутри агент может использовать JSON, trace, shell, pytest, policies,
evidence chains и logs.

Снаружи он должен говорить человеку ясно:

- как он понял задачу;
- что он собирается сделать;
- что уже сделал;
- что проверено;
- что не проверено;
- где нужен approval;
- сколько бюджета потрачено;
- почему он остановился.

Технический язык уместен для debug/report режима. В обычном операторском
режиме агент должен говорить человеческим языком.

## Бюджет

Бюджет нужен не только для денег. Бюджет ограничивает:

- количество autonomous cycles;
- количество LLM calls;
- количество web/search/fetch calls;
- количество test runs;
- время выполнения;
- количество approval requests;
- количество repair attempts;
- дневной и сессионный расход.

Если бюджет исчерпан, агент должен остановиться и объяснить причину.

Правильный поток:

```text
before action
  -> budget check
  -> allow or deny
  -> log decision
  -> if denied: circuit breaker stop
```

Агент не должен "просто так кушать ключи". Любой расход должен иметь цель,
лимит и audit trail.

## Approval

Approval нужен не для того, чтобы тормозить агента, а чтобы отделить безопасную
автономность от опасной.

Read-only и dry-run операции могут выполняться автоматически внутри бюджета.

Операции с риском требуют approval:

- изменение кода;
- изменение политики;
- включение внешнего канала;
- отправка сообщений наружу;
- финансовое действие;
- удаление данных;
- non-dry-run autonomous mode.

Если approval отсутствует, агент должен остановиться или положить запрос в
Approval Inbox.

## Runtime

Настоящая автономность начинается с runtime-слоя.

Runtime должен давать:

- task queue;
- scheduler;
- work session mode;
- budget governor;
- circuit breaker;
- approval inbox;
- health checks;
- recovery;
- periodic reports;
- safe dry-run mode by default.

Первый runtime должен быть скучным и безопасным: смотреть, проверять,
планировать, dry-run ingest, запускать tests, писать отчёт.

Только после стабильности можно разрешать ограниченные действия.

## Под-Агенты

Под-агенты не должны появляться как хаотичные копии главного агента.

Под-агент должен иметь:

- роль;
- область ответственности;
- доступные инструменты;
- бюджет;
- policy;
- memory scope;
- reporting contract;
- kill switch.

Главный агент остаётся orchestrator. Под-агенты исполняют ограниченные роли.

## Личный Рабочий Агент Сначала

Сначала агент должен стать полезным для владельца проекта.

Приоритеты:

- объяснять код;
- анализировать архитектуру;
- помогать с планом развития;
- запускать проверки;
- находить ошибки;
- предлагать исправления;
- учиться на проекте;
- готовить отчёты;
- тренироваться на задачах фриланса без реальных обязательств.

Продажа клиентам или внешний production режим появляются позже.

## Инварианты

Эти правила нельзя ломать ради скорости:

- evidence перед уверенностью;
- tests перед самоуверенностью;
- approval перед risky action;
- rollback перед write;
- budget перед loop;
- logs перед доверием;
- source ranking перед memory;
- human-readable report перед молчаливой автономией.

## Следующий Путь

Ближайшие правильные слои:

1. Autonomous Runtime dry-run.
2. Persistent Task Queue.
3. Scheduler / periodic work sessions.
4. Budget policy configuration.
5. Approval Inbox persistence.
6. Health monitor.
7. Bounded self-repair jobs.
8. Sub-agent manager.
9. External channel governance.
10. Production deployment boundary.

Коротко: автономный агент — это управляемая система развития. Его сила не в
том, что он "делает всё", а в том, что он умеет становиться способнее без
потери контроля, проверяемости и безопасности.
