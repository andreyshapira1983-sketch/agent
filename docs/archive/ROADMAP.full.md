# Дорожная карта

Идеи по развитию автономности, инструментов и интеграций. Приоритеты можно менять по обратной связи (см. [FEEDBACK.md](FEEDBACK.md)).

---

## Автономность и цикл

- **Расширение целей планировщика** — больше ключевых слов и шаблонов в `make_plan(goal)`, опционально генерация шагов через LLM по описанию цели.
- **Опциональное одобрение по типам инструментов** — в `autonomy_manager.needs_confirmation(action)` учитывать не только имя действия, но и уровень риска (например, всегда запрашивать подтверждение для `write_file` / `accept_patch` в определённых путях).
- **TTL для кэша fetch_url** — реализовано: `fetch_cache_ttl_sec` в конфиге; при 0 TTL не используется.

---

## Инструменты

- **manage_queue** — реализован: статус очереди, enqueue (task_id, tool, arguments), dequeue; интеграция с `tasks.queue`.
- **Алерты при превышении порога** — реализован инструмент **check_performance_alerts**: проверяет tool_times на превышение `warn_threshold_sec`, возвращает отчёт и пишет в audit `performance_alert` при обнаружении.
- **Новые инструменты** — по запросу: расчёт метрик по логам, чтение/запись в дополнительные хранилища с теми же правилами (лимиты, политика, метрики времени).
- **Улучшения кэша** — метрики попаданий/промахов реализованы: `get_fetch_cache_stats()` возвращает `fetch_cache_hits`, `fetch_cache_misses`. Инструмент **reset_fetch_cache** — сброс кэша по команде (реализован).

---

## Интеграции

- **Документирование подключённых систем** — в README и ARCHITECTURE уже добавлен краткий раздел о регистрации инструментов и внешних сервисах; при добавлении новых MCP или API — обновлять список и пример вызова.
- **Единый формат конфигурации ключей** — вынести перечень внешних сервисов (Telegram, OpenAI, Notion, …) и способ задания ключей в один раздел конфига или env-документации.

---

## Производительность и метрики

- **Пороговые значения** — реализовано: в конфиге `tool_performance.warn_threshold_sec`; при превышении времени выполнения инструмента — предупреждение в лог и запись в audit (`tool_slow`).
- **Анализ производительности инструментов** — утилита/инструмент `analyze_tool_performance` по данным `tool_times`: самые медленные инструменты, среднее время (реализовано в мониторинге).
- **Сводка по циклу** — после каждого автономного цикла логируется одна строка со временем выполнения инструментов (реализовано в orchestrator). Приоритизация оптимизаций — по данным: смотреть **analyze_tool_performance** и записи **tool_slow** в audit.
