## How to read this

- **Canonical ID:** `MIR-NNN`. Prior IDs from other documents are kept as
  **aliases** (never discarded).
- **Merge rule applied:** entries are merged **only** when they share a *proven
  common root* (same code location / same causal chain that the source docs
  themselves document). Similar symptoms alone were **not** merged. The five
  CORE learning-loop findings are a documented chain but live at independent
  locations with independent fix status, so they are kept as **separate issues**
  cross-linked under the "learning-loop cluster," not collapsed into one.
- **Provenance tag** per issue: `previously_documented` (already in a repo doc),
  `independently_reconfirmed` (re-verified by a later audit and by this pass),
  `newly_discovered` (first raised here).
- **Status vocabulary:** `open`, `partially_fixed`,
  `code_fixed_needs_runtime_verification`, `fixed`, `regression`, `duplicate`,
  `planned_gap`, `stale_finding`, `not_a_bug`, `needs_investigation`.

### MIR-002 — `_compute_quality_score` returns 1.0 on an empty evidence chain
- **Aliases:** CORE-03, LPF-011, A3. **Provenance:** previously_documented + independently_reconfirmed.
- **Sources:** LIVE_PROBE_FINDINGS.md, CORE_AUDIT_2026-07-18.md, MEMORY_FIX_PLAN.md.
- **First documented:** LIVE_PROBE_FINDINGS.md @ `e1d8081` (2026-07-18).
- **Files/functions:** `core/smart_memory.py` `_compute_quality_score` (`:51-52`).
- **Symptom:** a zero-evidence (pure general-knowledge / self-declared) answer banks `answer_quality_score=1.0`, contradicting a `fully_unverified` verifier report.
- **Root cause:** `if total == 0: return 1.0`.
- **Related causes:** couples with MIR-001 (outcome), MIR-023 (LPF-002: general knowledge has evidence=0). Same nerve.
- **Production path:** same as MIR-001 (`answer_quality_score` set in `episode_from_agent_cycle`).
- **Existing tests:** `tests/test_smart_memory.py` (asserts current 1.0 behaviour).
- _(SUPERSEDED — see this entry's Status/Resolution; kept as the record of what was considered)_ **Missing tests:** a regression asserting empty-chain → neutral floor (only if A3 is adopted).
- **Status:** **`fixed` (2026-07-20, commit `f16febf`)** ⬆ from `open`. Resolved by alternative (c) — separating "no evidence" from the quality axis — not by the A3 floor.
- **Resolution:** `_compute_quality_score` now returns **`None`** when there are no chunks: with an empty denominator the fraction is undefined, and the top of the scale is not a neutral value. `answer_quality_score` is `float | None`. Each of the four readers states its own treatment: staleness → mid-scale (`_QUALITY_WHEN_UNMEASURED`); pruning → candidate, tracked as `unmeasured` **separately** from `low_quality`; re-ask → lowers the threshold as for weak answers; fast path → refuses replay (`AgentLoop._quality_allows_replay`). The arbitrary-constant forks (a)/(b) were rejected: both keep reporting a measurement that was never taken.
- **Resolution evidence (measured before → after):** staleness groundless `1.33 → 2.08` vs well-evidenced `1.63` (higher = pruned sooner, so the inversion is gone); pruning at `min_quality=0.9` `['well-evidenced'] → ['evidenced','groundless']`; re-ask hint on a groundless answer `False → True`. Guard: measurable ratios unchanged (`8,2 → 0.8`). Tests: `tests/test_quality_score_unknown.py` (7, six fail-before). Full suite 4777 passed / 0 failed.
- **Evidence:** code still returns `1.0` on an empty chain; the docstring reframes it as "neutral." `_compute_quality_score(0,0,0)=1.0`, `(1,9,0)=0.1` (reproduced). **Downstream trace — the 1.0 is NOT inert:**
  - **`core/loop.py:757-790` `episodic_fast_path`** fires when a similar past episode has `answer_quality_score >= 0.70` **and** `not tools_used` **and** a non-empty `full_answer` (plus `similarity >= 0.85`). An empty-chain answer yields exactly `quality=1.0, outcome='success', tools_used=()`, so it **satisfies the gate**. **Reproduced:** even a *wrong* ungrounded answer ("Sydney" for Australia's capital) → `quality=1.0`, gate = `True`. Effect: on a later near-identical question the ungrounded answer is **replayed verbatim, the full verify pipeline is SKIPPED, and it is re-banked with `verified_chunks=1`** (bypasses a check **and** banks as verified success — exactly the two downstream effects that turn this from a design choice into a defect). The gate never consults the original verifier verdict, so a `fully_unverified` answer is replay-eligible (ties to MIR-016).
  - **`core/episodic_hygiene.py:125`** prunes on `answer_quality_score < min_quality`; a `1.0` ungrounded episode is protected from pruning (kept as high-value).
  - **`core/smart_memory.py:409` / re-ask detection** lowers the repeat-question threshold only when `quality < 0.5`; a `1.0` ungrounded answer suppresses the "user is re-asking because the last answer was weak" hint.
- **Scope/severity:** the fast-path also needs `similarity >= 0.85`, no file hint, no local-critique, non-`:` input — so the sharp case is *near-duplicate general-knowledge questions whose first answer was ungrounded/possibly wrong*. For a genuinely-correct general-knowledge answer, replay is fine; the defect is that the score cannot tell a correct from an incorrect ungrounded answer. Moderate severity, real path.
- **Blast-radius note (Stage-3.3):** ungrounded `quality=1.0` episodes come from more than general knowledge — the **verifier soft-fail** path (`core/loop.py:1533`, when the verifier itself throws) records `verified=0,unverified=0,fully_unverified=False` → a `1.0` episode. So a verifier *crash* also produces a fast-path-eligible "perfect" episode. Widens the case for fixing MIR-002 at the provenance level, not the score.
- _(SUPERSEDED — see this entry's Status/Resolution; kept as the record of what was considered)_ **Alternatives (still a real fork, but now bounded by the downstream):** (a) MEMORY_FIX_PLAN A3 floor (e.g. 0.3) for `total==0`; (b) keep 1.0 for the *score* but make `episodic_fast_path` additionally require the original episode passed verification (not just `quality>=0.70`); (c) separate a "no-evidence" flag from the quality axis so downstream gates can treat "confidently ungrounded" distinctly. Whichever is chosen must fix the **fast-path replay**, not just the number in isolation.
- _(SUPERSEDED — see this entry's Status/Resolution; kept as the record of what was considered)_ **Alternatives (undecided):** (a) apply A3 floor 0.3; (b) formally accept 1.0-as-neutral and close A3 as won't-fix; (c) separate "no-evidence" from "quality" so general knowledge isn't scored on an evidence axis at all. **Decision needed** — do not treat as a straightforward bug until the product decision is made.

### MIR-003 — one dubious success mints an `active` procedure, no quality gate
- **Aliases:** CORE-02, MGA-03, LPF-013, A4. **Provenance:** previously_documented + independently_reconfirmed.
- **Sources:** MEMORY_SYSTEM_AUDIT.md, CORE_AUDIT_2026-07-18.md, LIVE_PROBE_FINDINGS.md, MEMORY_FIX_PLAN.md.
- **First documented:** MEMORY_SYSTEM_AUDIT.md @ `accd746`.
- **Files/functions:** `core/smart_memory.py` `procedure_from_episode` (`:578`) → `ProceduralStore.upsert_from_episode` (`:464`); smoothing `_smoothed_confidence` (`:67`).
- **Symptom:** a single `success` episode with tools mints a procedure born `status="active"`; Beta(1,1) smoothing puts one success at 0.667 ≥ 0.6.
- **Root cause:** `procedure_from_episode` gates only on `outcome=="success" and tools_used`; no `answer_quality_score` threshold; born `active`.
- **Related causes:** upstream MIR-001 (outcome) and MIR-002 (quality).
- **Production path:** `_record_experience_memory` → `procedural_store.upsert_from_episode` → `procedure_from_episode`.
- **Existing tests:** `tests/test_smart_memory.py` (procedure creation).
- **Missing tests:** ~~a low-quality success does **not** create an `active` procedure~~ — the correctness half is now regression-locked (see the 2026-07-22 addition). The A4 *born-`candidate`* half stays untested because the behaviour it describes is not implemented (no `candidate` status exists).
- **Status:** **`fixed`** (2026-07-22) ⬆ from `code_fixed_needs_runtime_verification` ⬆ from `partially_fixed`. Correctness half regression-locked in #116; A4 maturity gate implemented in the follow-up; **now verified on a live agent against a real model** (see the Runtime verification note below). Both halves closed.
- **Evidence:** the direct gate is **unchanged** (still born `active`, no quality arg). **But** the MIR-001 fix reclassifies most low-quality answers as `partial` *before* they reach here, so the specific CORE-02 trigger is largely closed upstream. The `answer_quality_score<0.5` gate at `smart_memory.py:409` is in the **re-ask/Jaccard** path, **not** procedure creation — it does **not** address this issue. **Stage-3.3 reproduction:** a `verified=1,unverified=10` episode is now `partial` → `procedure_from_episode` returns **None** (no procedure). A genuine `verified=3,unverified=0` success → procedure still born `status='active'` (conf 0.5). So the exploit path is closed; the A4 "born `candidate` + quality threshold" design change is still not implemented (a single *genuine* success still activates immediately).
- **Approved fix (doc):** MEMORY_FIX_PLAN A4 (born `candidate` + quality threshold on birth) — **not applied**.
- **Addition (2026-07-22) — the correctness half is now proven and regression-locked; the design half is isolated and stated precisely.**
  - **Root re-verified as localized.** `ProcedureRecord` is constructed in exactly two places, both in `core/smart_memory.py` (`procedure_from_episode` birth `:1290`, `with_episode` update `:363`); the only external caller is `core/loop_methods2.py:654` `procedural_store.upsert_from_episode`, which routes through the single birth gate `procedure_from_episode` — no bypass. The decision lives in one file.
  - **Reproduced (read-only, in-process):** every non-completing episode mints **no** procedure — `verified=1,unverified=10` (`partial`), a grounded-but-`blocked` `verified=3` (`outcome=success` but `completion=blocked`), and a `refused` run all return `procedure_credit_allowed=False` → `procedure_from_episode=None`. Only a genuine `achieved` success mints one, born `active`. The "bad-reinforcement" defect this entry was opened for is therefore **closed**, by `procedure_credit_allowed` (MIR-057 completion axis) combined with MIR-001's outcome reclassification.
  - **Regression-locked** by `tests/test_smart_memory.py::test_grounded_but_incomplete_success_mints_no_procedure` and `::test_incomplete_success_mints_no_procedure_through_the_store`. These cover the previously-untested **completion axis** on procedure birth (a well-grounded `blocked`/`refused` answer — the exact MIR-057 live case), which the existing `failed`/`partial`/`weak` tests do not. **Fail-before demonstrated:** with the gate reduced to the pre-MIR-057 predicate (`outcome == "success" and tools_used`), the same blocked episode mints a `status="active"` procedure and both tests fail; on current code they pass. Full suite `5077 passed`.
  - **What is NOT fixed, stated exactly:** a single *genuine, verified, completed* success is still born `active` immediately (conf 0.5 at birth, 0.667 after `with_episode`). Making it born unproven (A4's *maturity gate*) requires a status value that **does not exist** — `ProcedureStatus = Literal["active","needs_review","obsolete"]` has no `candidate` (MEMORY_FIX_PLAN A4) or `extracted` (MEMORY_LIFECYCLE_CONTRACT, unapproved v2-draft). Adding one cascades to `_procedure_status` coercion (`:1371`), the consolidation status buckets (`:1308-1310`), and the retrieval path that injects `active` procedures into planning, and the two governing docs disagree on the name. **This is an owner architecture decision (status vocabulary), not a bounded bug fix, and is deliberately left unimplemented here.**
- **OWNER DECISION (2026-07-22) — the maturity gate is approved, and the status-vocabulary ambiguity is resolved.** The repository owner decided: *«A new procedure is born as `candidate` and does not participate in normal planning until confirmed by an independent, completed and verified execution.»*
  - **Name resolved:** `candidate` (MEMORY_FIX_PLAN A4), **not** `extracted` (the unapproved M1 contract). This is now the repository-owned policy the implementer may rely on.
  - **Two required behaviours:** (1) birth status = `candidate`, never `active`; (2) a `candidate` procedure is **excluded from ordinary planning retrieval** until it is promoted; (3) promotion to `active` requires a **second, independent, completed *and* verified** execution — i.e. `procedure_credit_allowed` must hold for a *distinct* confirming episode, not the birth episode.
  - **Framing:** this is the **next protective layer against premature learning**, not a fix of the original MIR-003 bug (that bug — dubious/incomplete reinforcement — is already closed and now regression-locked, above). It hardens the *genuine-single-success* case.
  - **Scope / sequencing:** implementation is a **separate follow-up PR**, not this one (this PR changes no behaviour). It will touch `ProcedureStatus` (add `candidate`), `procedure_from_episode` birth, `_procedure_status` coercion, the consolidation status buckets, and the retrieval filter that injects procedures into planning — plus fail-before tests that a one-success `candidate` is not planned with and is promoted only on a second independent completed+verified success. **When that lands, this entry moves toward `fixed`; until then it stays `partially_fixed`.**
- **IMPLEMENTED (2026-07-22, the follow-up PR) — `core/smart_memory.py` only.** The owner decision above is now code:
  - `ProcedureStatus` gains `candidate`; a shared `_procedure_status_for(success_count, confidence)` decides status in the two live setters (`with_episode`, `with_outcome`): `needs_review` if `confidence < 0.6` (demotion unchanged, checked first), else `candidate` while `success_count < _PROMOTION_MIN_SUCCESSES` (=2), else `active`.
  - Birth (`procedure_from_episode`) is `candidate`; a single credited success stays `candidate` (conf 0.667, which the old rule read as `active`).
  - **Planning exclusion:** `ProceduralMemoryStore.search` — the sole path that injects procedures into the planner (`core/loop_methods2.py:354`) — skips `candidate`. Promotion to `active` requires a **second, independent, completed+verified** success (`success_count` only rises through `procedure_credit_allowed`, MIR-057), after which it surfaces to planning.
  - `_procedure_status` coercion accepts `candidate`, so a stored candidate is not silently rewritten on load.
  - **End-to-end timeline (tested):** retrieval is at cycle start, promotion at cycle end, so a workflow is first *reused* on the third encounter (born c1, promoted c2, injected c3) — one cycle later than before, by design.
  - **Deliberately out of scope (stated):** the legacy-confidence migration (`recompute_legacy_confidence`, `:820`) and already-stored procedures keep the two-value rule — this decision governs *new* procedures; the consolidation report (a dead sink, MIR-044) is not expanded with a candidate bucket, so a candidate is counted in `procedure_count` but listed in no status bucket.
  - **Tests** (`tests/test_smart_memory.py`): 4 new (`born_candidate_not_active`, `candidate_not_offered_to_planning`, `second_independent_success_promotes_candidate_to_active`, `candidate_status_survives_serialization_round_trip`) + 3 updated to the new behaviour (`single_success_procedure_is_not_certain`, the injection timeline, the consolidation linkage). **Fail-before:** all four new tests and the updated single-success test fail on the pre-change code (`assert 'active' == 'candidate'`). Full suite `5081 passed`.

### MIR-048 — procedural memory has no reachable negative feedback (one-way confidence ratchet)
- **Aliases:** none. **Related:** MIR-003 (birth gate — different root: birth vs demotion), MIR-044. **Provenance:** newly_discovered (M1 review, operator-challenge follow-up).
- **Files/functions:** sole production path `core/loop_methods2.py:364` → `upsert_from_episode` (`core/smart_memory.py:464-467`) → `procedure_from_episode` (`:578-580`) returns `None` for any `outcome != "success"` — so `with_episode` (`:221-240`) never receives a non-success episode. The `failure_count` increment (`:224`) is **dead code in production**; no other writer exists.
- **Symptom:** procedure confidence `(s+1)/(s+2)` rises monotonically from 0.667; `needs_review` (`<0.6`) is unreachable via outcomes; **a procedure can fail forever and never demote**. Consolidation's `needs_review` list is structurally near-empty (compounds MIR-044's emptiness).
- **Status:** **`fixed` (2026-07-20)** ⬆ from `open`. Demotion is now reachable through outcomes.
- **Resolution:** `ProceduralMemoryStore.apply_episode_feedback(episode)` consumes **only** `used_procedure_ids` (MIR-049). No fallback exists — not workflow_key, tool set, name similarity, a fresh retrieval, or inference from content — because MIR-050 measured that keys pool unrelated goals, so any of those would debit an uninvolved procedure. An unresolvable id is reported `orphaned`, never substituted. Outcome matrix: `success` → `success_count+1`; `partial` and `failed` → `failure_count+1`; **cancelled → nothing** (a control signal is not evidence a procedure is bad); `used_procedure_ids` `None` (legacy) and `()` (known-none) are both no-ops and stay distinguishable in the report.
- **Atomicity:** `ProcedureRecord.with_outcome` recomputes counter, confidence and status into one new record, so no reader can see a bumped counter beside a stale confidence.
- **Idempotence:** keyed on the episode id already present in the procedure's own `source_episode_ids` journal — an explicit lookup, not a guess from the counters' shape. Re-processing a queue cannot turn the ratchet twice. Wired after `upsert_from_episode`, which journals the same id on a fresh success, so the two paths compose without double-counting.
- **Tests:** `tests/test_procedure_outcome_feedback.py` — 15 passing, all fail-before. Central case continues the MIR-049 scenario to its conclusion: two procedures sharing a workflow_key, only one attributed, run failed → the attributed one is debited and the other comes out **byte-identical**. Plus the outcome matrix, cancellation as a no-op, no-attribution no-ops, orphaned ids, re-application, multi-procedure (one discrete observation each, not a split share), duplicate collapsing, partial-orphan resilience, and `demotion_is_now_reachable` — four attributed failures move a procedure to `needs_review`.
- **Sequencing (operator decision, 2026-07-20):** attribution must land **before** feedback. Without `used_procedure_ids`, the only way to decide *which* procedure a failed cycle should debit is to re-derive it from `workflow_key` or a similar heuristic — and MIR-050 has now measured that those keys pool unrelated goals (8/48 keys span multiple goal classes). The result would be: failed episode → approximately-matched procedure → `failure_count` incremented on the **wrong** procedure. That is worse than the current ratchet, because a one-way ratchet is inert while misattributed punishment is actively wrong. Exact causal link first, counter updates second.
- **Measured scale (live store, read-only, 2026-07-20):** `failure_count == 0` on **all 65** procedures — the ratchet is not theoretical, it has never once turned. 64/65 are `active`; 53/65 rest on a single success. Non-success episodes do exist (e.g. `partial×2 + success×1` under `tools:shell_exec`), they simply never reach a procedure.
- **Root formula impact:** any lifecycle driver built on these counts (incl. §6.4 of the draft M1) inherits a positive-only signal — this is why "B-enhanced" alone is not a root fix.

### MIR-049 — no usage→outcome attribution for procedures (write-only `_last_procedure_records`)
- **Aliases:** none. **Related:** MIR-048 (feedback), MIR-050 (identity) — separate roots, kept apart. **Provenance:** newly_discovered (M1 review).
- **Files/functions:** `core/loop.py:392`, `core/loop_methods2.py:229-230` — procedures injected into planning are recorded in `_last_procedure_records`, which is **read nowhere**; banked episodes carry no `used_procedure_ids`.
- **Symptom:** a cycle that consumed a procedure and then failed/produced an unverified answer never updates that procedure; the only update channel is coincidental (a later success sharing the same tool sequence). Reputation without attribution.
- **Status:** **`fixed` (2026-07-20)** ⬆ from `open`. Observation only — counters deliberately untouched, so "is the attribution correct" stays provable separately from "is the feedback policy correct" (MIR-048).
- **Resolution:** `EpisodeRecord.used_procedure_ids: tuple[str, ...] | None` with **three** distinguishable states — `None` = legacy row, attribution unknown and nothing may be inferred; `()` = this version ran and is certain no procedure applied; `(ids…)` = application observed. `resolve_used_procedures(selected, executed_tools)` requires **two** gates: the procedure was selected into the run (retrieval alone is not use) **and** every tool in its workflow actually executed. Judged from execution rather than the plan, so a run cancelled before reaching a procedure's steps never debits it. `AgentLoop._executed_tools` accumulates as steps complete, so an exception cannot discard attribution already earned. Order is first actual completion; the result is a tuple, never a set, so the stored record is reproducible.
- **Central invariant (executable):** two procedures sharing one `workflow_key`, only one selected → the episode names only that one. This is the direct proof that MIR-049 removes the risk MIR-050 measured: attribution follows the selection, never the shared shape.
- **Tests:** `tests/test_used_procedure_attribution.py` — 12 passing, all fail-before. Matrix: retrieved-but-not-selected, selected-but-not-reached, partial execution, full execution, repeat use recorded once, two procedures in deterministic order, attribution surviving `partial`/`failed`, cancellation after application, replay attributing nothing, legacy reading as `None` not `()`, and a dangling id never re-pointed via `workflow_key`.
- **Prerequisite satisfied for MIR-048:** a trustworthy causal link now exists to consume.
- **Why it must precede MIR-048:** MIR-048 needs to know which procedure a failed cycle should debit. The only alternative to a recorded `used_procedure_ids` is re-deriving the link from `workflow_key`, which MIR-050 has now measured to pool unrelated goals. Closing the feedback loop on top of an approximate link would replace an inert one-way ratchet with active misattribution — the system would honestly count failures and charge them to the wrong procedure.
- **Acceptance:** an episode records the procedures that were actually injected into its planning; a procedure's counters change only through that recorded link (or the repaired workflow-match channel), never through a re-derived guess.

### MIR-050 — procedure identity is tool-sequence-only; evidence aggregates across unrelated goals
- **Aliases:** none. **Related:** MIR-048/049 (they interact: attribution onto a meaningless aggregate is still meaningless). **Provenance:** newly_discovered (M1 review).
- **Files/functions:** `core/smart_memory.py:581` `workflow_key = "tools:" + "->".join(tools)`.
- **Symptom:** episodes with the same tool sequence but unrelated goals upsert into one procedure ("Workflow using file_read"); its success counts and confidence describe a semantically meaningless aggregate, so any lifecycle judgment over it is unfounded.
- **Status:** `open` — **harm confirmed, fix deliberately DEFERRED** (2026-07-20). ⬆ from `needs_investigation`: the defined measurement ran, so this is no longer a mechanism without evidence. It is deferred on the merits, not forgotten — see the return condition below.
- **Measurement (live stores, read-only, 2026-07-20; nothing mutated):**
  - 65 procedures / 65 distinct `workflow_key`s; 200 episodes, 68 carrying tools, across 48 distinct tool-sequences.
  - **Keys spanning more than one goal-signature class: 8 of 48 (17%).**
  - **Episodes whose evidence is pooled across goal classes: 27 of 68 (40%).**
  - **Worst collision:** `tools:file_read->list_dir` — 10 distinct goal classes over 10 episodes, i.e. every episode under that key is a different goal.
  - **Outcome mixing under one key (the concrete wrong judgment):** `tools:shell_exec` holds `partial×2 + success×1` from three different goals — one procedure's counters aggregate the success of one task with the failures of others.

Only then is D-1 decidable.
- **Note:** fix remains a design decision (key = tools + goal-class/token signature, or per-goal sub-buckets) — Open Decision D-1 in `MEMORY_LIFECYCLE_CONTRACT.md` §16, still not prejudged.

Before it was formally registered, that number was mistakenly taken by a different finding — retrieval observability — in commit `fca27ed`. Declining, being blocked by a budget or a permission, and stopping part-way all present identically to the outcome derivation, which is why the entry is scoped to the class rather than to refusals.
- **Symptom (live store, read-only, verified in the data rather than quoted from a log):** the **only** episode ever admitted to steer later answers is a blocked non-completion.

Nothing in the derivation asks whether the goal was achieved. Verification measures whether claims are grounded; it cannot measure whether the question was answered. A well-cited non-completion scores well **because** it honestly cited why it could not finish. Two different questions were folded into one field, and the field is named after the one it does not measure.

- **Status:** `open` (newly_discovered). Registration and analysis only — no code changed, no live store touched.
- **Why the marker scan is not the fix:** the keyword scan above is a **sizing instrument, not a proposed mechanism**. It can over- and under-count, and shipping it as a gate would replace a wrong signal with an unreliable one.
- **There is no completion signal in the system today.** This must not be understated into "a signal exists and is not consulted". What exists are detectors of *particular kinds of evidence shortfall* — `evaluate_low_evidence_policy` (`core/low_evidence_policy.py:210`), the `⚠️ Не подтверждено` note (`core/loop_helpers.py:356`), the `Unverified:` section marker (`core/loop_helpers.py:125`). Treating these as a completion signal would repeat this defect one layer up: a second evidence measure wearing a completion label. **A structured completion signal has to be produced, not discovered.**

  1. the live blocked episode above does not bank `success`, and its procedure credit does not occur;

  4. `test_non_completion_does_not_credit_a_procedure` — `success_count` byte-identical after the cycle;
  5. `test_non_completion_is_neutral_for_an_uninvolved_procedure` — no debit either, when the procedure was not causally responsible;
  6. `test_a_non_completion_is_not_usage_eligible` — admission refuses it even if outcome is forced;
  7. `test_a_complete_evidenced_answer_is_still_success` — guard against over-correction;
  8. `test_a_lesson_stays_retrievable_without_becoming_a_replayable_success` — the `lesson` clarification: still surfaced as a lesson, not admitted as a proven workflow, and no positive procedural feedback;

### MIR-059 — a working binary is swapped for a broken one, and the arguments are not translated with it
- **Aliases:** none. **Related:** MIR-010 (found in the same live cycle; that entry made the failure VISIBLE, this one is why it happened at all). **Provenance:** newly_discovered (2026-07-20, live run).
- **Files/functions:** `tools/shell_exec.py` `_platform_alias` applied unconditionally at the dispatch site.
- **Symptom:** a cycle asked for the line count of `core/loop.py`, the planner produced `grep -c ^ core/loop.py`, and the answer came back `blocked` — for an environmental reason with nothing to do with the question. The tool had swapped `grep` for `findstr` even though real grep was installed, and then handed it a path it cannot parse.
- **Root cause, two independent faults:**
  1. **The swap was unconditional.** `_platform_alias` mapped `grep`->`findstr` on Windows whether or not the requested binary existed. Measured: `shutil.which("grep")` resolves to Git's `usr/bin/grep.EXE`, and that binary answers `grep -c ^ core/loop.py` with `4093` using **either** slash style. A working tool was replaced by a failing one.
  2. **Only the NAME was translated, never the arguments.** `findstr` reads `/` as a switch prefix, so `core/loop.py` parses as `core` plus `/l /o /o /p` — hence `FINDSTR: Cannot open loop.py`. Reproduced directly: the same command with a backslash path returns 0.
- **Status:** **`fixed` (2026-07-20)**.
- **Resolution:** `_resolve_binary` prefers the requested command whenever it is on PATH and falls back to the platform equivalent only when it is absent, returning `(name, substituted)`. `_normalise_argv_for` rewrites path separators **only when a substitution actually happened** — the requested binary already understands its own dialect, and rewriting for it would be damage. The output records `executed_command` and `binary_substituted`, because the live log showed `argv: ['grep', ...]` beside `FINDSTR:` in stderr with nothing connecting them, and a substitution nobody can see is one nobody can debug.
- **Flags are deliberately NOT translated.** `grep -c` has no findstr equivalent worth guessing at, and a half-built dialect mapper turns every unmapped flag into a new silent failure. After MIR-010 a fallback that cannot read its arguments fails visibly, which leaves the planner able to see it and re-plan.
- **Live verification:** the exact failing invocation now returns `executed_command=grep, substituted=False, exit_code=0, stdout='4093', execution_status=success` — where it previously produced `blocked`.
- **Tests:** `tests/test_shell_binary_resolution.py` — 8 passing, 7 fail-before. Covers an installed binary never being substituted, the equivalent used only as a fallback, an absent pair still raising, separators rewritten only on substitution, argv byte-identical without one, flags passed through untranslated, the executed binary recorded, and the live invocation end to end.
- **Residual:** resolution depends on PATH, so the same argv can reach different binaries from different shells. That is now honest (use what is installed) rather than silently wrong (always swap), but it stays environment-dependent and worth remembering when reading a log.

- **Aliases:** none. **Related:** MIR-057 (its forward policy stopped the accrual this entry is about), MIR-051 (same store, different defect — that was an invariant violation in a derived field, this is standing earned under a superseded rule), MIR-050 (identity, which bounds what any repair could mean). **Provenance:** newly_discovered (2026-07-20, measured while building the MIR-057 legacy report).

- **Symptom:** MIR-057's gates are forward-only. They stopped new credit; they did not unwind credit already granted. **47 of 65 procedures hold non-zero counters together with source references to episodes that carry no completion verdict** — aggregate `success=72, failure=0`. Those procedures are `active` (46) or `needs_review` (1), and are still ranked by confidence and injected into planning: the retrieval path reads `confidence`/`status`, not completion, because a procedure is not an episode and has no completion axis of its own.
- **What is proven and what is not — stated precisely because the distinction is the whole entry:** `source_episode_ids` is an applied-feedback journal. - **Status:** `open` (newly_discovered). Diagnosed only; no repair attempted and none designed.
- **Why it is not part of MIR-057:** distinct root (data written under a superseded rule vs. a policy that measured the wrong thing), distinct blast radius (65 stored procedures vs. every future cycle), distinct remediation (a decision about existing standing vs. a gate), and distinct completion criteria. Folding it in would let a data question ride out on a logic fix.
- **Evidence:** `python scripts/completion_legacy_report.py` — read-only, verifies its own sha256 before and after, and prints the per-procedure legacy-source coverage, the confidence distribution and the status distribution.

After a 14-cycle live run the same command reported **36 of 65, aggregate success=61** — not because standing was verified, but because the episodic store's 200-record FIFO window evicted the referenced episodes, so their references now read as `missing` and drop out of the count. Do not decide this without also deciding MIR-050.
- **Missing tests:** whichever option is chosen needs a fail-before test that the affected procedures no longer present unverifiable standing as verified, plus a guard that verifiable standing is untouched.

### MIR-060 — verification confirms that the source exists, never that the derived claim follows from it
- **Aliases:** none. **Related:** MIR-046 (**same line of code, different axis** — see the no-merge basis below), MIR-042 (what may enter the chain), MIR-002 (the ratio these verdicts feed), MIR-057 (the other axis `outcome` cannot express). **Provenance:** newly_discovered (2026-07-20, operator instruction during the structured-outputs capability spike).
- **Files/functions:** `core/verifier_core.py:120-153` — `ev = match_citation(c, chain)`; if it resolves and `strict_ok` survives, `any_matched = True` → `verdict = "verified"`. Content gates in that branch are exactly two: `_memory_citation_is_independent` (trust class, MIR-046's fix) and a literal figure-containment test gated on `is_statistical_claim` (`core/verifier_utils.py:59-69`, `core/verifier_patterns.py:49-72`).
- **Symptom:** a claim whose truth is a **computation over** the cited excerpt — a count, a sum, a difference, a ratio, a comparison, an inference — is marked `verified` because the citation resolved. The verifier never evaluates the claim against the excerpt, so it cannot distinguish a correct derivation from a false one.
- **Measured, in-memory, read-only (2026-07-20).** Synthetic chain, one `file` evidence with excerpt `"alpha=1\nbeta=2\ngamma=3\n"`, `verify(expects_contract_headers=False)`:

  | claim (all citing `[file:vals.txt]`) | true? | verdict |
  |---|---|---|
  | "The three values sum to 6" | yes | `verified` |
  | "The three values sum to 99" | **no** | `verified` |
  | "The file defines 7 keys" | **no** | `verified` |
  | "gamma is smaller than alpha" | **no** | `verified` |
  | "The file contains beta=2" | yes (literal) | `verified` |
  | "Coverage rose 40%" | **no** | `topic_supported_but_claim_unverified` |
  | "The average value is 12" | **no** | `topic_supported_but_claim_unverified` |

  Four false derived claims score **identically to the true one and to the literal quotation**. Only the last two differ, and only because `_STAT_TRIGGER_RE` fired on `%` / `average`.
- **The one band that does bite is containment, not evaluation.** Re-running "The average value is 12" against an excerpt extended with an unrelated `port=12` flips the verdict from `topic_supported_but_claim_unverified` to **`verified`**. The test asks whether the figure's characters appear somewhere in the excerpt; it does not compute the average, and it has no notion of which number belongs to which quantity. Its trigger set is also narrow — `%`, currency, `N x`, numeric ranges bearing a time unit, and the words `most`/`majority`/`average` (and Russian equivalents). A count, a sum, a line number, a latency in `ms`, a version comparison — none of them trigger it at all.
- **The semantic check is never consulted once a citation resolves.** `_find_semantic_support` (NLI) and `_find_structured_support` are reachable **only** from the branch where nothing matched (`core/verifier_core.py:163-188`). Measured directly: verifying "The three values sum to 99" with an LLM stub that answers `NO` to every NLI question yields `verified` and **0 NLI calls**. Resolution short-circuits evaluation — the machinery that could catch this exists in the same file and is structurally unreachable for exactly the claims that need it.
- **Why this is NOT part of MIR-046 (no-merge basis).** Same line, orthogonal axes. MIR-046 asked *whose* evidence may count and fixed it by adding `Evidence.origin` and demoting non-`user-explicit` memory. This entry asks *whether the cited content supports the claim* and is untouched by that fix — every case measured above uses a `file` evidence that MIR-046 considers fully independent. The connection is real and worth recording: MIR-046's own symptom line already noted "verified iff the citation resolves to a record in the chain, never because the content was checked", and then scoped its fix to the trust axis. **The content axis was observed there and never registered.** Distinct root, distinct fix (a claim-vs-excerpt entailment judgement vs. a provenance field), distinct closure criteria. Folding them together would let a fixed entry appear to cover an open one.
- **Blast radius.** `verified_chunks` is not a display value. It drives `_compute_quality_score` (MIR-002), the `outcome` derivation, `decide_usage_eligibility` (`verified_chunks > 0` is an admission condition), procedural credit, and the confidence gate. A false derived claim therefore does not merely reach the user — it can make its episode look well-supported enough to be admitted as reusable experience. The MIR-057 completion axis is an **independent** guard: it asks whether the goal was reached, not whether a claim was computed correctly, so a completed task resting on a false derivation passes both.
- **Operator-cited observations (supplied, not independently located):** responses **184 / 182 / 172** were cited as showing the verifier confirming the presence of a source while the computed assertion went unchecked. Those response records were **not found in this repository** — no `docs/`, `data/` or `logs/` artifact matches those identifiers. The mechanism above is measured and stands on its own; the three responses would ground it in live output and should be attached when their location is known.
- **Structured Outputs do not fix this, and must not be argued as if they might.** A JSON schema constrains the *shape* of what the synthesizer emits. This defect is in what the verifier *checks* about a claim's content, downstream of any transport. The two are independent; see `docs/audit/PROVIDER_STRUCTURED_OUTPUT_AUDIT.md` §8.
- **Architectural placement (added 2026-07-21; the defect above is unchanged by it).** The check this entry finds missing is named in the target architecture as §5 *Tool Result Validation → Semantic Validation*, and in §8 as *Evidence Consistency Check* / *Factuality Evaluation*. Recorded as orientation only: per that document's own reading rule a named component means "foreseen", not "implemented", and naming a slot neither explains the defect nor discharges it. **This is a reference, not a reclassification — MIR-060 stays a reproducible present defect with the measurements above, not "an unbuilt slot".**
- **The M1 memory-lifecycle contract does NOT close this (measured against the draft, 2026-07-21).** Its verification rules — `docs/audit/MEMORY_LIFECYCLE_CONTRACT.md` §7.4, §12, §14 — are entirely **trust-class** rules: a resolved citation counts `verified` unless the matched evidence's `trust_class` is `agent_derived` or `replay_echo`, and §7.4 states in terms that "citation resolution keeps exactly its current meaning". A `file`-kind evidence derives `trust_class = tool_observation` (§4.3), which is inside the allowed set, so **every case in the table above would still count `verified` under that contract as drafted**. This is not a fault in the contract — its §1 scopes it to record trust, the same axis as MIR-046 — but it must be recorded, because the contract is the canonical M1 design and planning from it would otherwise assume the content axis is covered. ### MIR-061 — one malformed memory record silently drops every record after it from the provenance chain
- **Aliases:** none. **Related:** MIR-042 (**same two lines of code, different root** — the trust-filter face), MIR-052 (same anti-pattern family: a broad `except` whose comment describes behaviour the code does not have), MIR-060 (a truncated chain is one way a resolvable citation stops resolving). **Provenance:** newly_discovered (2026-07-21); **re-registered 2026-07-22** — first landed with its fix in #114, both reverted by #115, now restored on its own so the fix travels without the reverted bundle.
- **Files/functions:** `core/loop.py` persistent-record chain injection. The `try` wraps the **entire `for` loop**; the handler is a bare `pass`.
- **Symptom:** a single record that raises while being converted to evidence aborts the whole injection loop, so **every record after it is silently omitted from the provenance chain**. The verifier then evaluates the answer against an incomplete chain.
- **The code's own comment states the opposite behaviour.** It reads *"A malformed record stays out of the chain but the loop completes normally"* — singular, and describing a per-record skip. The loop does not complete; it is abandoned.
- **Measured (2026-07-21, in-process, read-only), using the exact control flow of both sites** with five records of which the **second** is malformed: the persistent-record site (`try` outside the loop) delivered **1 of 5**; the adjacent working-artifact site (`try` inside the loop) delivered **4 of 5**. Two sites doing the same job, on adjacent lines, with opposite granularity — and the one that is wrong is the one whose comment claims to be right.
- **Root cause:** exception scope. Not the presence of the guard (chain assembly genuinely must not abort a run) and not the trust model (that is MIR-042) — simply a `try` placed around the loop instead of around the iteration.
- **Consequence:** the verifier's verdict is computed against whatever reached the chain. Citations that *should* resolve then come back `cited_but_unmatched`, so the answer is marked unverified — or its disclaimer fires — for a reason that has nothing to do with the answer or its sources. Because the handler is a bare `pass`, nothing records that the chain was truncated; the resulting verdict looks like an ordinary evidence shortfall.
- **Why this is NOT part of MIR-042 (no-merge basis).** Same two lines, orthogonal roots. MIR-042 asks *which records may enter* the chain (trust filter on `Evidence.origin`); this asks *whether the records that may enter actually arrive* (fixed by moving the `try` inside the loop). Distinct blast radius, distinct closure criteria, distinct urgency. Folding them would let a data-loss bug ride out on a trust-model fix.
- **Status:** `code_fixed_needs_runtime_verification` (2026-07-22). Fixed by moving the `try` inside the loop and logging `memory_evidence_skipped` instead of passing silently; live-agent runtime with a real model not exercised.
- **Resolution evidence:** `tests/test_evidence_integration.py::TestMalformedMemoryRecordDoesNotTruncateChain` — `test_one_bad_record_only_skips_itself` (N records, one malformed → N−1 reach the chain, only the bad one dropped) and `test_skipped_record_is_reported` (the drop is logged, not swallowed). **Fail-before verified:** both fail on the pre-fix `try`-around-the-loop code; both pass after. A guard is added that the two injection sites keep the same granularity so they cannot drift apart again.
- **History note:** the identical fix first shipped in PR #114 (commit `667dd6b`) and was reverted with the rest of #114 by PR #115 for reasons unrelated to this defect; this entry and PR restore it in isolation.

### MIR-051 — stored procedure `confidence` violates the Beta-smoothing invariant (legacy data)

Writes go through the store's own writer, so the integrity envelope is recomputed normally. Fixtures are seeded **through the store**, so every case exercises the real integrity-envelope row shape rather than a convenient in-memory object. Covers the four correction scenarios, already-consistent byte-identity, idempotence, counters/freshness/identity/content preservation, invalid counters skipped rather than guessed, dry-run writing nothing, and a bounded resumable pass.
- **Not addressed here (by design):** the missing failure history itself — no run reconstructs failures that were never recorded. That is MIR-048, and it is sequenced behind MIR-049.
- **Invariant to restore:** `stored_confidence == _smoothed_confidence(success_count, failure_count)` for every procedure whose counters are valid.
- **Migration requirements (all mandatory):** deterministic; idempotent (a second run is a no-op); available as `dry_run` first; bounded in volume or safely resumable; emits a before/after report; **does not alter `success_count` / `failure_count`**; **does not touch `updated_at`, `last_used_at` or any other freshness field** — a migration must not make stale procedures look recently used and thereby extend their life past hygiene. Recompute every field that is a pure derivative of the same counters (e.g. `status`, which is `confidence >= 0.6`), but **do not reconstruct missing failure history** — that is MIR-048's job, not a migration's.

### Fix-design contract (MIR-002 + MIR-041) — operator-approved target semantics

> **Superseded as the canonical source:** the single canonical lifecycle contract is now
> `docs/audit/MEMORY_LIFECYCLE_CONTRACT.md` (M1). This section remains as the historical
> defect-level record; where the two differ, the M1 contract wins.

> Chosen approach: **Variant 2 + 3** (gate the fast-path on genuine verification **and** never
> record a replay as verified). **Variant 1 (arbitrarily lowering `quality` to 0.3) is rejected
> as the primary fix** — a quality *score* and a *proof-of-verification provenance* are different
> concepts and must not be conflated. Design MIR-002 and MIR-041 fixes **together** (same
> execution path) but keep them as **separate defects/commits**. Do NOT implement yet — finish
> the read-only memory trace first, then propose the provenance scheme + fail-before tests.

1. **Fast-path is allowed ONLY for an episode carrying explicit proof that the original answer
   actually passed Verify or rested on valid evidence.**
2. **Verification must NOT be inferred indirectly from** `answer_quality_score`, `outcome=="success"`,
   absence of `tools_used`, or `verified_chunks` **when that field was synthetically set by the
   fast-path itself.**
3. **A memory replay must be stored with its own provenance** (`source=episodic_fast_path` /
   `memory_replay`), and **must not receive `verified_chunks=1`** nor count as a new verification.
4. **A replay must not raise trust in the source episode** and must not create a self-sustaining
   verified-success chain.
5. **MIR-002 and MIR-041 stay separate defects**, fixed on the same path but not merged.
6. **No arbitrary `quality→0.3`** as the main fix; quality score ≠ verification provenance.

**Provenance scheme — operator-corrected, TWO INDEPENDENT AXES (do not merge origin and verification into one field):**

- **Axis 1 — `verification_status`**: `verified` | `partially_verified` | `unverified` | `verifier_error` | `not_run`.
  Set from the **actual verifier outcome only**. MUST NOT be inferred from `answer_quality_score`, `outcome`,
  `tools_used`, `source_labels`, or old-record `verified_chunks`.
- **Axis 2 — `response_origin`**: `generated` | `memory_replay` | `persistent_memory` | (any existing origin already in the system).

- **Memory replay must be stored as:** `response_origin = memory_replay`; `verification_status = not_run`;
  keep `source_episode_id` (or equivalent link to the source episode); `verified_chunks = 0`; the replay
  creates **no new evidence**; and the replay episode **must not** become a candidate for the next fast-path.
- **Fast-path requires:** source episode `verification_status == verified` **AND** the record is **not** a replay
  **AND** all existing safe constraints (no file hint, non-`:`, not local-critique, `full_answer` present,
  `not tools_used`) still hold.
- **Old episodes without `verification_status` → treat as `not_run`** and **disallow fast-path**. **No automatic
  migration by guessing** from old fields.

**MIR-002 is NOT fully fixed by this.** This closes only its **fast-path manifestation** (an ungrounded/`verifier_error`
`quality=1.0` episode will no longer be replay-eligible, because the gate now keys on `verification_status==verified`,
not the score). The influence of `quality=1.0` on **hygiene pruning, re-ask detection, and the other readers** remains
a **separate open part of MIR-002**.

**Post-trace deliverables (before any code):** [reader list — DONE, in AUDIT_PROGRESS] · **separate** fail-before tests
for MIR-002 and MIR-041 (approved next step) · then the minimal file set for the fix.

### MIR-010 — command failure recorded as step success
- **Aliases:** CORE-06, LPF-010. **Related (broader):** OFM-004 (MIR-031). **Provenance:** previously_documented + independently_reconfirmed.
- **Sources:** LIVE_PROBE_FINDINGS.md, CORE_AUDIT_2026-07-18.md, OPERATIONAL_FAILURE_MODES.md.
- **Files/functions:** `tools/shell_exec.py` `validate_output` (`:520-539`); `core/loop.py` step gate (`~:2985/:3301`).
- **Symptom:** `where ffmpeg` returns `exit_code=1` but `tool_result.status="success"` and the step is treated as succeeding.
- **Root cause:** `shell_exec` sets `status="success"` whenever the subprocess *ran*; `exit_code` lives in `output` and no post-condition reads it; `validate_output` only warns on timed_out/exit_code *consistency*.
- **Related causes:** external analogue OFM-004 (MIR-031).
- **Production path:** AgentLoop act → `shell_exec` → step success gate → episode outcome.
- **Existing tests:** `tests/test_shell_exec*` (existing behaviour).
- **Missing tests:** per-command exit-code semantics (`exit!=0` → failure for run-commands; `1` still "not found" for `where`/`grep`/`test`).
- **Status:** **`fixed` (2026-07-20)** ⬆ from `open`. The design decision the entry demanded was taken on measured data.
- **Live confirmation before the fix:** a real cycle ran `grep -c ^ core/loop.py`, got `exit_code=1` with `stderr="FINDSTR: Cannot open loop.py"`, and the log carried `tool_result status=success` beside `verify ok=True`. The command could not open the file and every layer above it said otherwise.
- **Why the blanket rule stayed rejected — measured, not argued:** `grep` found → 0/empty stderr; no match → **1/empty stderr** (a legitimate ANSWER); missing file → 2/non-empty. But `where` not-found → **1 with NON-empty stderr** (`ИНФОРМАЦИЯ: не удается найти файлы`), and `which` not-found → 1 with `which: no zzz in (...)`. So both a blanket `exit!=0→failure` AND a family-blind "stderr means trouble" heuristic would recast an ordinary negative result as an error.
- **Resolution:** `classify_shell_result` in `tools/shell_exec.py` returns **two separate facts** — `execution_status` (success/failure: did the command run correctly) and `answer_result` (positive/negative/not_applicable: what it answered). "grep found nothing" is a successful execution with a negative answer; conflating that with "grep could not open the file" is what produced the defect. Per-family contracts: `grep`/`egrep`/`fgrep` and `findstr` read stderr at exit 1; `rg`, `diff`/`cmp`, `test`/`[`, `where`, `which` key on the exit code alone; anything else takes the conservative unknown contract (0 → success, non-zero → failure); `exit_code is None` (timeout) → failure.
- **Classified on the binary that ACTUALLY RAN.** `_platform_alias` swaps `grep`→`findstr` on Windows, so reading the requested name would check one tool's exit code against another tool's contract. `git` deliberately takes the unknown contract: it is a multiplexer whose semantics differ per subcommand, and guessing a family from `argv[1]` is the inference this design refuses.
- **The three layers stay separate.** `run()` returns the raw `exit_code`/`stdout`/`stderr` untouched plus the two normalised fields; `validate_output` still answers only "does this match the schema", because a failed command produces a perfectly well-formed report OF that failure and rejecting it would conflate malformed output with a correct failure message; a new overridable `Tool.execution_status` hook lifts the verdict into the top-level `ToolResult.status`.
- **`answer_result` is telemetry about one command** and is explicitly NOT a completion signal — it says nothing about task completion, episodic eligibility or procedural feedback (MIR-057 owns those).
- **Tests:** `tests/test_shell_exit_code_semantics.py` — 47 passing. Every row of the contract table, plus an integration pair through `Tool.invoke`: a real failure (`git rev-parse --verify zzznosuchref` → exit 128) that was `status=success`/`ok=True` before and is `execution_status=failure`/`status=error` after, and a real negative answer that must stay `success`. Reproduced on the old code by stashing `tools/`.
- **Not fixed here (separate root):** on Windows the alias translates the NAME but not the ARGUMENTS — FINDSTR reads `/` as a switch prefix, so `core/loop.py` becomes `core` plus `/l /o /o /p`, and POSIX flags like `-c` do not exist. That is why the live `grep` failed at all. Registered separately; a path/flag adapter must not ride out on an exit-code fix.

### MIR-040 — `recover_stuck` is defined and unit-tested but never wired to a consumer
- **Aliases:** none (relates to OFM-009 recovery; daemon-progress item 4.2 "Recovery on start"). **Provenance:** newly_discovered (Stage 3).
- **Sources:** this pass.
- **Files/functions:** `core/task_queue.py:329` `recover_stuck` (resets `running`→`pending` after `timeout_minutes`); tested in `tests/test_task_queue.py`; **zero production callers** (`core/`, `app/`, `cli/`, `main.py`, `agent_tick.py`).
- **Symptom:** a task left in `running` by a crashed process (see MIR-039) is never auto-recovered in the live system, because nothing calls `recover_stuck`.
- **Root cause:** the recovery mechanism exists but is not invoked by any runtime entry point (self-audit-lessons anti-pattern #6: "a module written to fix a live failure mode, never wired into its entry point").
- **Related causes:** the stuck state is produced by MIR-039; the always-on drain that would call it does not exist (MIR-033); startup recovery is explicitly `daemon-progress.md` item **4.2 "Recovery on start" — not started**.
- **Production path:** none — the function is unreachable from any live consumer.
- **Existing tests:** `tests/test_task_queue.py` (unit-tests the function in isolation).
- **Status:** `open` (an attempted one-line wiring was made and then found **unsafe** — see below; it must NOT be treated as a valid fix). ⬇ downgraded from an incorrect `fixed` after a safety review.
- **Attempted fix (commit `02a8a58` on `fix/mir-040-wire-recover-stuck`) — UNSAFE, DO NOT MERGE:** `run_task_queue` was made to call `task_queue.recover_stuck()` at entry. A behavioral test (`test_run_task_queue_recovers_stuck_running_task`) proved fail-before / pass-after and 177 related tests stayed green — but that only proved recovery *happens*, not that it is *safe*. A follow-up safety review reproduced two real hazards this wiring introduces:
  - **Double execution under concurrent consumers + long tasks (regression risk).** `recover_stuck` uses a `timeout_minutes` (default 30) staleness on `updated_at`, and `updated_at` is set once at `mark_running` and **never refreshed during a run** (verified: no heartbeat inside the `run_task_queue` loop). So a *live* task legitimately running longer than the timeout is indistinguishable from a crashed one. Reproduced (isolated tmp): a task `running` for 40 min is reset to `pending` by a second consumer's `recover_stuck` → becomes eligible → a second process claims and runs the **same task concurrently**. Before this change stuck tasks were merely *stranded* (safe-but-useless); after it they can be *double-run* (unsafe). The consumer entry points (`app/task_scheduler_cli.py` `:task-run`, `agent_tick.py`) hold **no single-instance lock** (`app/single_instance.py` exists but is not acquired here), so concurrent consumers are genuinely possible.
  - **`max_attempts` cap bypass.** `recover_stuck` resets status to `pending` **without touching `attempts`**. Reproduced: a task stuck at `attempts == max_attempts` (e.g. an OOM-killer task) is resurrected to `pending`, then re-claimed → `attempts` becomes `max_attempts + 1`, running one time past the cap. On the recovery path a deterministically process-killing task is retried rather than being finalized `failed` — a mild resurgence of the OFM-010 class.
  - **Real process crash not exercised.** Both the original test and the hazard repros *simulate* a crash (`mark_running` + backdated `updated_at`); an actual killed process leaving a stuck task and a real subsequent recovery was **not** run.
- **Correct design (for a real fix — larger than a one-liner):** recovery must run **only at startup under the single-instance lock** (`app/single_instance.py`; this is `daemon-progress.md` item **4.2 "Recovery on start" — not started**), when no task is genuinely in-flight; **and/or** add a task heartbeat that refreshes `updated_at` during a run so staleness reflects liveness; **and/or** record owner PID/host and verify the owner is dead before reclaiming; **and** `recover_stuck` must respect `attempts`/`max_attempts` (a stuck-exhausted task → terminal `failed`, not resurrected). None of these is the ad-hoc `run_task_queue` wiring.
- **Resolution (operator-approved):** the unsafe wiring was **reverted** — commit `e8d330d` on `fix/mir-040-wire-recover-stuck` reverts `02a8a58`; `core/autonomous_runtime.py` and `tests/test_autonomous_runtime.py` now match `main` (verified: no diff), `recover_stuck` has **no production caller** again (safe-but-stranded behavior restored), and 44 task_queue/autonomous_runtime tests pass. MIR-040 stays **`open`**; the real fix is the guarded startup-recovery above (daemon 4.2), to be scheduled as its own work — not this ad-hoc wiring.
- **Note on docs:** OPERATIONAL_FAILURE_MODES.md §5 calls `recover_stuck()` a "verified control present"; it remains **unwired in a safe way** — refresh that doc's wording in a later pass (see MIR-036).
