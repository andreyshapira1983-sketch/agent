### MIR-002 — `_compute_quality_score` returns 1.0 on an empty evidence chain
- **Aliases:** CORE-03, LPF-011, A3. **Provenance:** previously_documented + independently_reconfirmed.
- **Sources:** LIVE_PROBE_FINDINGS.md, CORE_AUDIT_2026-07-18.md, MEMORY_FIX_PLAN.md.
- **First documented:** LIVE_PROBE_FINDINGS.md @ `e1d8081` (2026-07-18).
- **Files/functions:** `core/smart_memory.py` `_compute_quality_score` (`:51-52`).
- **Symptom:** a zero-evidence (pure general-knowledge / self-declared) answer banks `answer_quality_score=1.0`, contradicting a `fully_unverified` verifier report.
- **Root cause:** `if total == 0: return 1.0`.
- **Related causes:** couples with MIR-001 (outcome), MIR-023 (LPF-002: general knowledge has evidence=0). Same nerve.
- **Production path:** same as MIR-001 (`answer_quality_score` set in `episode_from_agent_cycle`).
- **Existing tests:** `tests/test_smart_memory.py` (asserts current 1.0 behaviour).
- _(SUPERSEDED — see this entry's Status/Resolution; kept as the record of what was considered)_ **Missing tests:** a regression asserting empty-chain → neutral floor (only if A3 is adopted).
- **Status:** **`fixed` (2026-07-20, commit `f16febf`)** ⬆ from `open`. Resolved by alternative (c) — separating "no evidence" from the quality axis — not by the A3 floor.
- **Resolution:** `_compute_quality_score` now returns **`None`** when there are no chunks: with an empty denominator the fraction is undefined, and the top of the scale is not a neutral value. `answer_quality_score` is `float | None`. Each of the four readers states its own treatment: staleness → mid-scale (`_QUALITY_WHEN_UNMEASURED`); pruning → candidate, tracked as `unmeasured` **separately** from `low_quality`; re-ask → lowers the threshold as for weak answers; fast path → refuses replay (`AgentLoop._quality_allows_replay`). The arbitrary-constant forks (a)/(b) were rejected: both keep reporting a measurement that was never taken.
- **Resolution evidence (measured before → after):** staleness groundless `1.33 → 2.08` vs well-evidenced `1.63` (higher = pruned sooner, so the inversion is gone); pruning at `min_quality=0.9` `['well-evidenced'] → ['evidenced','groundless']`; re-ask hint on a groundless answer `False → True`. Guard: measurable ratios unchanged (`8,2 → 0.8`). Tests: `tests/test_quality_score_unknown.py` (7, six fail-before). Full suite 4777 passed / 0 failed.
- **Evidence:** code still returns `1.0` on an empty chain; the docstring reframes it as "neutral." `_compute_quality_score(0,0,0)=1.0`, `(1,9,0)=0.1` (reproduced). **Downstream trace — the 1.0 is NOT inert:**
  - **`core/loop.py:757-790` `episodic_fast_path`** fires when a similar past episode has `answer_quality_score >= 0.70` **and** `not tools_used` **and** a non-empty `full_answer` (plus `similarity >= 0.85`). An empty-chain answer yields exactly `quality=1.0, outcome='success', tools_used=()`, so it **satisfies the gate**. **Reproduced:** even a *wrong* ungrounded answer ("Sydney" for Australia's capital) → `quality=1.0`, gate = `True`. Effect: on a later near-identical question the ungrounded answer is **replayed verbatim, the full verify pipeline is SKIPPED, and it is re-banked with `verified_chunks=1`** (bypasses a check **and** banks as verified success — exactly the two downstream effects that turn this from a design choice into a defect). The gate never consults the original verifier verdict, so a `fully_unverified` answer is replay-eligible (ties to MIR-016).
  - **`core/episodic_hygiene.py:125`** prunes on `answer_quality_score < min_quality`; a `1.0` ungrounded episode is protected from pruning (kept as high-value).
  - **`core/smart_memory.py:409` / re-ask detection** lowers the repeat-question threshold only when `quality < 0.5`; a `1.0` ungrounded answer suppresses the "user is re-asking because the last answer was weak" hint.
- **Scope/severity:** the fast-path also needs `similarity >= 0.85`, no file hint, no local-critique, non-`:` input — so the sharp case is *near-duplicate general-knowledge questions whose first answer was ungrounded/possibly wrong*. For a genuinely-correct general-knowledge answer, replay is fine; the defect is that the score cannot tell a correct from an incorrect ungrounded answer. Moderate severity, real path.
- **Blast-radius note (Stage-3.3):** ungrounded `quality=1.0` episodes come from more than general knowledge — the **verifier soft-fail** path (`core/loop.py:1533`, when the verifier itself throws) records `verified=0,unverified=0,fully_unverified=False` → a `1.0` episode. So a verifier *crash* also produces a fast-path-eligible "perfect" episode. Widens the case for fixing MIR-002 at the provenance level, not the score.
- _(SUPERSEDED — see this entry's Status/Resolution; kept as the record of what was considered)_ **Alternatives (still a real fork, but now bounded by the downstream):** (a) MEMORY_FIX_PLAN A3 floor (e.g. 0.3) for `total==0`; (b) keep 1.0 for the *score* but make `episodic_fast_path` additionally require the original episode passed verification (not just `quality>=0.70`); (c) separate a "no-evidence" flag from the quality axis so downstream gates can treat "confidently ungrounded" distinctly. Whichever is chosen must fix the **fast-path replay**, not just the number in isolation.
- _(SUPERSEDED — see this entry's Status/Resolution; kept as the record of what was considered)_ **Alternatives (undecided):** (a) apply A3 floor 0.3; (b) formally accept 1.0-as-neutral and close A3 as won't-fix; (c) separate "no-evidence" from "quality" so general knowledge isn't scored on an evidence axis at all. **Decision needed** — do not treat as a straightforward bug until the product decision is made.

### MIR-048 — procedural memory has no reachable negative feedback (one-way confidence ratchet)
- **Aliases:** none. **Related:** MIR-003 (birth gate — different root: birth vs demotion), MIR-044. **Provenance:** newly_discovered (M1 review, operator-challenge follow-up).
- **Files/functions:** sole production path `core/loop_methods2.py:364` → `upsert_from_episode` (`core/smart_memory.py:464-467`) → `procedure_from_episode` (`:578-580`) returns `None` for any `outcome != "success"` — so `with_episode` (`:221-240`) never receives a non-success episode. The `failure_count` increment (`:224`) is **dead code in production**; no other writer exists.
- **Symptom:** procedure confidence `(s+1)/(s+2)` rises monotonically from 0.667; `needs_review` (`<0.6`) is unreachable via outcomes; **a procedure can fail forever and never demote**. Consolidation's `needs_review` list is structurally near-empty (compounds MIR-044's emptiness).
- **Status:** **`fixed` (2026-07-20)** ⬆ from `open`. Demotion is now reachable through outcomes.
- **Resolution:** `ProceduralMemoryStore.apply_episode_feedback(episode)` consumes **only** `used_procedure_ids` (MIR-049). No fallback exists — not workflow_key, tool set, name similarity, a fresh retrieval, or inference from content — because MIR-050 measured that keys pool unrelated goals, so any of those would debit an uninvolved procedure. An unresolvable id is reported `orphaned`, never substituted. Outcome matrix: `success` → `success_count+1`; `partial` and `failed` → `failure_count+1`; **cancelled → nothing** (a control signal is not evidence a procedure is bad); `used_procedure_ids` `None` (legacy) and `()` (known-none) are both no-ops and stay distinguishable in the report.
- **Atomicity:** `ProcedureRecord.with_outcome` recomputes counter, confidence and status into one new record, so no reader can see a bumped counter beside a stale confidence.
- **Idempotence:** keyed on the episode id already present in the procedure's own `source_episode_ids` journal — an explicit lookup, not a guess from the counters' shape. Re-processing a queue cannot turn the ratchet twice. Wired after `upsert_from_episode`, which journals the same id on a fresh success, so the two paths compose without double-counting.
- **Tests:** `tests/test_procedure_outcome_feedback.py` — 15 passing, all fail-before. Central case continues the MIR-049 scenario to its conclusion: two procedures sharing a workflow_key, only one attributed, run failed → the attributed one is debited and the other comes out **byte-identical**. Plus the outcome matrix, cancellation as a no-op, no-attribution no-ops, orphaned ids, re-application, multi-procedure (one discrete observation each, not a split share), duplicate collapsing, partial-orphan resilience, and `demotion_is_now_reachable` — four attributed failures move a procedure to `needs_review`.
- **Sequencing (operator decision, 2026-07-20):** attribution must land **before** feedback. Without `used_procedure_ids`, the only way to decide *which* procedure a failed cycle should debit is to re-derive it from `workflow_key` or a similar heuristic — and MIR-050 has now measured that those keys pool unrelated goals (8/48 keys span multiple goal classes). The result would be: failed episode → approximately-matched procedure → `failure_count` incremented on the **wrong** procedure. - **Root formula impact:** any lifecycle driver built on these counts (incl. §6.4 of the draft M1) inherits a positive-only signal — this is why "B-enhanced" alone is not a root fix.

The only alternative to a recorded `used_procedure_ids` is re-deriving the link from `workflow_key`, which MIR-050 has now measured to pool unrelated goals. Closing the feedback loop on top of an approximate link would replace an inert one-way ratchet with active misattribution — the system would honestly count failures and charge them to the wrong procedure.
- **Acceptance:** an episode records the procedures that were actually injected into its planning; a procedure's counters change only through that recorded link (or the repaired workflow-match channel), never through a re-derived guess.

### Fix-design contract (MIR-002 + MIR-041) — operator-approved target semantics

> **Superseded as the canonical source:** the single canonical lifecycle contract is now
> `docs/audit/MEMORY_LIFECYCLE_CONTRACT.md` (M1). This section remains as the historical
> defect-level record; where the two differ, the M1 contract wins.

> Chosen approach: **Variant 2 + 3** (gate the fast-path on genuine verification **and** never
> record a replay as verified). **Variant 1 (arbitrarily lowering `quality` to 0.3) is rejected
> as the primary fix** — a quality *score* and a *proof-of-verification provenance* are different
> concepts and must not be conflated. Design MIR-002 and MIR-041 fixes **together** (same
> execution path) but keep them as **separate defects/commits**. Do NOT implement yet — finish
> the read-only memory trace first, then propose the provenance scheme + fail-before tests.

1. **Fast-path is allowed ONLY for an episode carrying explicit proof that the original answer
   actually passed Verify or rested on valid evidence.**
2. **Verification must NOT be inferred indirectly from** `answer_quality_score`, `outcome=="success"`,
   absence of `tools_used`, or `verified_chunks` **when that field was synthetically set by the
   fast-path itself.**
3. **A memory replay must be stored with its own provenance** (`source=episodic_fast_path` /
   `memory_replay`), and **must not receive `verified_chunks=1`** nor count as a new verification.
4. **A replay must not raise trust in the source episode** and must not create a self-sustaining
   verified-success chain.
5. **MIR-002 and MIR-041 stay separate defects**, fixed on the same path but not merged.
6. **No arbitrary `quality→0.3`** as the main fix; quality score ≠ verification provenance.

**Provenance scheme — operator-corrected, TWO INDEPENDENT AXES (do not merge origin and verification into one field):**

- **Axis 1 — `verification_status`**: `verified` | `partially_verified` | `unverified` | `verifier_error` | `not_run`.
  Set from the **actual verifier outcome only**. MUST NOT be inferred from `answer_quality_score`, `outcome`,
  `tools_used`, `source_labels`, or old-record `verified_chunks`.
- **Axis 2 — `response_origin`**: `generated` | `memory_replay` | `persistent_memory` | (any existing origin already in the system).

- **Memory replay must be stored as:** `response_origin = memory_replay`; `verification_status = not_run`;
  keep `source_episode_id` (or equivalent link to the source episode); `verified_chunks = 0`; the replay
  creates **no new evidence**; and the replay episode **must not** become a candidate for the next fast-path.
- **Fast-path requires:** source episode `verification_status == verified` **AND** the record is **not** a replay
  **AND** all existing safe constraints (no file hint, non-`:`, not local-critique, `full_answer` present,
  `not tools_used`) still hold.
- **Old episodes without `verification_status` → treat as `not_run`** and **disallow fast-path**. **No automatic
  migration by guessing** from old fields.

**MIR-002 is NOT fully fixed by this.** This closes only its **fast-path manifestation** (an ungrounded/`verifier_error`
`quality=1.0` episode will no longer be replay-eligible, because the gate now keys on `verification_status==verified`,
not the score). The influence of `quality=1.0` on **hygiene pruning, re-ask detection, and the other readers** remains
a **separate open part of MIR-002**.

**Post-trace deliverables (before any code):** [reader list — DONE, in AUDIT_PROGRESS] · **separate** fail-before tests
for MIR-002 and MIR-041 (approved next step) · then the minimal file set for the fix.

### MIR-042 — memory records enter the provenance chain with no trust-class filter (semantic-side laundering seam)
- **Aliases:** none. **Related:** MIR-041 (episodic analogue), MIR-009 (missing trust/temporal model), MIR-002. **Provenance:** newly_discovered (Stage-3.3 whole-subsystem trace).
- **Sources:** this pass.
- **Files/functions:** `core/loop.py:1305-1318` (persistent records → `chain.add(evidence_from_memory_record(...))` for every retrieved record) and `:1331-1347` (working-memory artifacts → `chain.add(make_evidence(kind="memory", ...))`); `core/evidence.py:584 evidence_from_memory_record`.
- **Symptom / mechanism:** every retrieved persistent/working memory record is injected into the verifier's provenance chain **uniformly** — the record's `source` (user-explicit vs agent-auto) is read (`getattr(rec,"source",None)`) but **not used to gate or trust-tag** the evidence. A `[memory:<id>]` citation therefore resolves to a *matched* evidence. So a stored fact — including one written by a prior, possibly-unverified answer — can back a later answer as if it were checked evidence.
- **Why it matters (ties to the operator's semantics rule #2):** this is the semantic-memory analogue of MIR-041 — "cited a memory record" is being treated as support without regard to whether the record's content was ever verified. - **Update (2026-07-20):** the *counting* face is closed by MIR-046 (`5105e03`): a resolved `[memory:]` citation to an `agent-auto` record no longer contributes `verified`. The *injection* face remains open — `loop.py:1305/:1331` still adds every retrieved record to the chain with no trust filter. The consequence is now bounded (presence in the chain no longer buys a verified verdict) but the chain itself is still trust-blind, which matters for any future consumer that reads it directly.
- **Evidence (M0):** `core/verifier_core.py:87-116` — a citation that `match_citation`s **any** chain evidence, including `kind="memory"`, is marked `[verified:…]` and increments `verified_chunks` (line 114-116). Persistent records + working artifacts are injected into the chain uniformly with **no trust-class filter** (`loop.py:1305-1347`). So a `[memory:<id>]` citation to a stored record counts as *verified* — it verifies the citation *resolves to a real record*, not that the record's *content* was checked. **Severity raised:** `knowledge_auto_write` defaults to **True** in production (`app/bootstrap.py:250`), so agent-generated claims are auto-written to persistent memory and can later be cited as "verified." ### MIR-040 — `recover_stuck` is defined and unit-tested but never wired to a consumer
- **Aliases:** none (relates to OFM-009 recovery; daemon-progress item 4.2 "Recovery on start"). **Provenance:** newly_discovered (Stage 3).
- **Sources:** this pass.
- **Files/functions:** `core/task_queue.py:329` `recover_stuck` (resets `running`→`pending` after `timeout_minutes`); tested in `tests/test_task_queue.py`; **zero production callers** (`core/`, `app/`, `cli/`, `main.py`, `agent_tick.py`).
- **Symptom:** a task left in `running` by a crashed process (see MIR-039) is never auto-recovered in the live system, because nothing calls `recover_stuck`.
- **Root cause:** the recovery mechanism exists but is not invoked by any runtime entry point (self-audit-lessons anti-pattern #6: "a module written to fix a live failure mode, never wired into its entry point").
- **Related causes:** the stuck state is produced by MIR-039; the always-on drain that would call it does not exist (MIR-033); startup recovery is explicitly `daemon-progress.md` item **4.2 "Recovery on start" — not started**.
- **Production path:** none — the function is unreachable from any live consumer.
- **Existing tests:** `tests/test_task_queue.py` (unit-tests the function in isolation).
- **Status:** `open` (an attempted one-line wiring was made and then found **unsafe** — see below; it must NOT be treated as a valid fix). ⬇ downgraded from an incorrect `fixed` after a safety review.
- **Attempted fix (commit `02a8a58` on `fix/mir-040-wire-recover-stuck`) — UNSAFE, DO NOT MERGE:** `run_task_queue` was made to call `task_queue.recover_stuck()` at entry. A behavioral test (`test_run_task_queue_recovers_stuck_running_task`) proved fail-before / pass-after and 177 related tests stayed green — but that only proved recovery *happens*, not that it is *safe*. A follow-up safety review reproduced two real hazards this wiring introduces:
  - **Double execution under concurrent consumers + long tasks (regression risk).** `recover_stuck` uses a `timeout_minutes` (default 30) staleness on `updated_at`, and `updated_at` is set once at `mark_running` and **never refreshed during a run** (verified: no heartbeat inside the `run_task_queue` loop). So a *live* task legitimately running longer than the timeout is indistinguishable from a crashed one. Reproduced (isolated tmp): a task `running` for 40 min is reset to `pending` by a second consumer's `recover_stuck` → becomes eligible → a second process claims and runs the **same task concurrently**. Before this change stuck tasks were merely *stranded* (safe-but-useless); after it they can be *double-run* (unsafe). The consumer entry points (`app/task_scheduler_cli.py` `:task-run`, `agent_tick.py`) hold **no single-instance lock** (`app/single_instance.py` exists but is not acquired here), so concurrent consumers are genuinely possible.
  - **`max_attempts` cap bypass.** `recover_stuck` resets status to `pending` **without touching `attempts`**. Reproduced: a task stuck at `attempts == max_attempts` (e.g. an OOM-killer task) is resurrected to `pending`, then re-claimed → `attempts` becomes `max_attempts + 1`, running one time past the cap. On the recovery path a deterministically process-killing task is retried rather than being finalized `failed` — a mild resurgence of the OFM-010 class.
  - **Real process crash not exercised.** Both the original test and the hazard repros *simulate* a crash (`mark_running` + backdated `updated_at`); an actual killed process leaving a stuck task and a real subsequent recovery was **not** run.

- **Resolution (operator-approved):** the unsafe wiring was **reverted** — commit `e8d330d` on `fix/mir-040-wire-recover-stuck` reverts `02a8a58`; `core/autonomous_runtime.py` and `tests/test_autonomous_runtime.py` now match `main` (verified: no diff), `recover_stuck` has **no production caller** again (safe-but-stranded behavior restored), and 44 task_queue/autonomous_runtime tests pass.
