# Что сделано для агента и будет ли это работать

Кратко: что добавлено, куда подключено, и при каких условиях это реально работает (а не «для красоты»).

---

## Последние улучшения (Telegram, Planner Loop, инструменты)

- **Telegram:** Очередь по чату — сообщения в одном чате обрабатываются по одному, второе не теряется. Ошибки показываются коротко и по-русски (таймаут, сеть, доступ и т.д.). Команды `/help` (список команд) и `/cancel` (остановка автономного режима).
- **Planner Loop:** По команде `/stop` или `/cancel` автономный цикл останавливается не только после цикла, но и прерывает текущие задачи в цикле. Опционально «тихие часы»: в `config/agent.json` или env `QUIET_HOURS=23-7` — в это время автономные циклы пропускаются.
- **Новые инструменты:** `open_in_browser(url)` — открыть ссылку в браузере; `add_reminder(text, when_str)` — напоминание (завтра 18:00, через 2 часа и т.д.). Просроченные напоминания приходят в Telegram при следующем сообщении.
- **Команда /remind:** например `/remind завтра 18:00 купить молоко` или `/remind через 2 часа позвонить`.
- **Рефакторинг:** общий вызов обработчика и ответа в Telegram вынесен в `_run_handler_with_reply` (напоминания, понятные ошибки, один код для текста/фото/видео/голоса).

## Архитектура: Telegram как интерфейс, агент на ПК

**Основная идея:** Telegram не выполняет действия сам. Он только принимает сообщения, показывает результаты и передаёт их агенту. Вся реальная работа (PowerShell, Python, файлы, программы) делается на компьютере, где запущен агент.

| Вариант | Где работает | Что делает |
|--------|---------------|------------|
| Telegram-only | В облаке/на сервере | Только отвечает на сообщения, не управляет ПК |
| **Telegram + Local Agent** | **На твоём компьютере** | **Агент получает команды через Telegram, выполняет действия на ПК** ← наш случай |
| Fully autonomous local | На ПК | Сам планирует, Telegram только уведомления |

**Схема:**

```
+-----------------+
| Telegram Client |  (телефон / браузер)
+--------+--------+
         |
         v
+-----------------+
| Telegram API    |
+--------+--------+
         |
         v
+-----------------+
| Local Agent     |  <-- работает на твоём ПК (python src/main.py)
| - Planner Loop  |      сам решает, что делать
| - Memory        |
| - LLM           |
| - Tools         |  <-- PowerShell, Python, файлы, pip, fetch...
+--------+--------+
         |
         v
+-----------------+
| OS / программы  |  (Windows, процессы, диск)
+-----------------+
```

**Вывод:** Агент знает, что Telegram — канал связи; его задача — обработать сообщения и при необходимости выполнить что-то на ПК. Он может работать в фоне (daemon): проверять Telegram, планировать свои действия (Planner Loop), выполнять задачи. Ты можешь общаться через Telegram, но агент не обязан ждать команды — он сам планирует и может слать уведомления.

## Как агент «работает на компьютере», когда ты пишешь в Telegram

**Вопрос:** он же в Telegram — как он тогда что-то делает на компе? В фоне или как?

**Ответ:** Бот — это обычная программа (Python), которую ты запускаешь **на своём компьютере** (`python src/main.py` или как у тебя настроено). Она подключается к Telegram и ждёт сообщения. Как только ты что-то пишешь в Telegram:

1. Сообщение приходит в **этот же процесс на твоём компе** (long polling или webhook).
2. Код вызывает агента (LLM + инструменты). LLM решает, что делать, и вызывает инструменты: `run_python`, `run_powershell`, `read_file`, `write_file` и т.д.
3. Эти инструменты выполняются **в том же процессе или в дочерних процессах (subprocess) на твоём компе**. То есть PowerShell или Python запускаются у тебя локально, но **без видимого окна** — в фоне, вывод перехватывается и передаётся обратно агенту.
4. Агент формирует ответ и отправляет его тебе в Telegram.

Итого: **«работа на компе» = выполнение кода на той машине, где запущен бот.** Если бот запущен у тебя на ПК — всё крутится у тебя. Окно PowerShell/Python по умолчанию не открывается (фоновый запуск); если нужно именно открыть программу или окно, агент может выполнить, например, `Start-Process notepad` или `Start-Process powershell` — тогда пользователь увидит окно.

## 14. Проактивный агент (сам пишет первым)

**Сделано:**  
- `src/communication/proactive_planner.py`:  
  - Лимиты: минимальный интервал (по умолчанию 6 ч) и макс. число сообщений в день (по умолчанию 3). Настраиваются через `TELEGRAM_PROACTIVE_MIN_INTERVAL_SEC`, `TELEGRAM_PROACTIVE_MAX_PER_DAY`.  
  - Состояние в `data/proactive_state.json`.  
  - `get_proactive_message()`: проверяет лимиты; затем пытается взять последнюю запись из лога прочитанного и сформировать сообщение («Недавно прочитал: …»); если лог пустой — один короткий запрос к LLM («одно предложение для разговора или NO»).  
  - `try_send_proactive()`: вызывает `get_proactive_message()`, при наличии текста шлёт его через `send_alert()` и обновляет состояние.  
- В `src/communication/autonomous_mode.py`: после каждого `run_cycle()` вызывается `try_send_proactive()`.  
- В `telegram_alerts.py`: `get_alerts_chat_id()`, `send_proactive_voice(voice_path)` для будущей отправки голосовых (сейчас планировщик шлёт только текст).

**Будет ли работать:** Да, но с условиями.  
- Проактивные сообщения уходят только когда крутится автономный цикл (после команды `/autonomous`). Без запущенного автономного режима проактивность не сработает.  
- Нужен чат для отправки: либо `TELEGRAM_ALERTS_CHAT_ID` в .env, либо хотя бы одно сообщение пользователя боту (тогда используется сохранённый chat_id).  
- Интервал и лимит в день реально ограничивают частоту; при пустом логе прочитанного и отказе LLM что-то предложить сообщение не отправится.

## Итог: что точно работает «в коде»

- **pip_install, run_python, run_powershell** — выполняются на машине с ботом при вызове агентом.  
- **fetch_url** по разрешённым сайтам (в т.ч. библиотеки), **log_reading**, **get_reading_log** — запись и чтение лога работают.  
- **search_openlibrary**, **get_gutenberg_book_list** — отдают списки книг; дальнейшее сканирование зависит от того, что агент по ним вызывает fetch_url и log_reading.  
- **Intent interpreter** — вызывается, подставляет [Intent: …] или запускает цикл; поведение агента по Intent зависит от следования промпту.  
- **Фото** — скачиваются, описываются через Vision, агент получает текст.  
- **Видео/голос** — уже были; промпт дополнен, чтобы не отказываться от ответа по описанию/расшифровке.  
- **Проактивные сообщения** — отправляются только при работающем автономном цикле и при соблюдении лимитов; текст формируется из лога или LLM.

Что зависит от поведения модели (промпт): выполнение «иди читать» без лишних уточнений, отказ подставлять метрики вместо ответа, честное «не смог прочитать», автономный выбор задачи на чтение. Всё это запрограммировано и подключено; будет ли агент вести себя стабильно — зависит от модели и контекста.

## Проактивный автономный агент: цели и Planner Loop

Реализован цикл **Observe → Think (LLM) → Select action → Execute → Update memory**, чтобы агент сам выбирал следующее действие по внутренним целям, а не только по командам.

### Внутренние цели (Goal Manager)

Список в `src/planning/planner_loop.py` — **INTERNAL_GOALS**:

| action_id          | Смысл |
|--------------------|--------|
| read_books         | читать книги, обучаться |
| analyze_system     | анализировать систему, улучшения |
| share_with_user    | поделиться с пользователем (проактивное сообщение) |
| run_maintenance    | метрики, здоровье системы |
| continue_queue     | fallback: выбор по эмоциям/правилам (task_creator) |

### Память и состояние (Observe)

В **think** передаётся краткое состояние: метрики (calls, successes, errors), self_assessment (success_rate), наличие inbox, последняя запись из лога прочитанного. См. `_state_summary_for_think(state)`.

### Planner Loop (когда очередь пуста)

1. **Observe** — уже есть: `state = self.observe()` в `run_cycle`.
2. **Think** — `think_next_action(state)`: один вызов LLM с состоянием и списком целей; ответ строго JSON `{action, reason}`.
3. **Select action** — разбор ответа; если `action` не из списка — используется `continue_queue`.
4. **Execute** — по выбранному действию:
   - `read_books` → в очередь: `get_gutenberg_book_list(5)`;
   - `analyze_system` → `analyze_self_model`, `get_improvement_plan`;
   - `share_with_user` → сразу вызывается `try_send_proactive()` (проактивное сообщение), в очередь ничего не ставится;
   - `run_maintenance` → `get_metrics`, `check_performance_alerts`;
   - `continue_queue` → дальше как раньше: `try_generate_and_enqueue(state)` (эмоции + хотелки).
5. В Telegram уходит короткое уведомление: «Мысль: \<reason\>. Делаю: \<action_id\>.»
6. **Update memory** — результаты выполнения попадают в лог, метрики, reflection как и раньше.

### Где подключено

- **Оркестратор** (`src/tools/orchestrator.py`, `run_cycle`): при пустой очереди сначала вызывается `think_next_action(state)`; при действии не `continue_queue` — `enqueue_planner_action()` и/или `try_send_proactive()`; иначе — старый `try_generate_and_enqueue(state)`.

### Будет ли работать

- **Да**: при пустой очереди вызывается LLM, ответ парсится, задачи ставятся в очередь или отправляется проактивное сообщение; уведомление «Мысль: …» уходит в чат.
- Зависит от модели: насколько стабильно она возвращает допустимый `action` и короткий `reason`; при сбое парсинга или пустом ответе срабатывает fallback `continue_queue` и эмоциональная генерация задач.
