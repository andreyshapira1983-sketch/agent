# Архитектура агента: кто принимает решения

## Проблема: LLM управляет системой

Если цикл выглядит так:

```
пользователь → LLM → решение → действие → LLM → решение → действие → …
```

то **агент не думает**: он только исполняет то, что сгенерировала модель. LLM не имеет полного состояния системы, памяти об изменениях и понимания архитектуры. Для модели правка кода — это редактирование текста; для системы — риск удаления файлов и поломки проекта.

Итог: **90% LLM, 10% агент** → нестабильно.

---

## Целевая схема: агент первый, LLM — только генератор

Правильный порядок:

```
цикл
  ↓
локальный анализ состояния (агент)
  ↓
решение (агент: что делать, какой файл, какая цель)
  ↓
если нужно изменение кода → один вызов LLM: «сгенерируй патч»
  ↓
валидация патча (агент: не пусто, не >30% удаления)
  ↓
действие (агент: propose_file_edit / write_file после подтверждения)
```

То есть **AGENT → (иногда) LLM**, а не наоборот.

- **LLM** используется только в одной роли: **генерация содержимого патча** (новое содержимое файла по цели).
- **Агент** делает: анализ, планирование, проверку, применение, откат.

## Где что реализовано

| Слой | Сейчас | Цель |
|------|--------|------|
| **Автономный цикл** (orchestrator) | `reason()` и `plan()` — локальные эвристики, без LLM. `act()` выполняет инструменты по плану. | Оставить как есть: цикл уже агентный. |
| **Диалог** (Telegram → ответ) | LLM получает все инструменты и в цикле сама решает: read_file → write_file → … | Ограничить: для правок кода использовать один защищённый инструмент. |
| **Правки кода из диалога** | LLM напрямую вызывает `write_file` / `propose_file_edit`. | Заменить на один поток: агент читает файл, один вызов LLM «только новый контент», агент проверяет и вызывает `propose_file_edit`. |

## Видимость workspace: агент должен видеть реальные файлы

Если агент не видит, какие файлы и каталоги реально существуют, он начинает угадывать пути и «слепнет» архитектурно. Это ломает базовый принцип: сначала локальный анализ состояния, потом решение.

Правильная схема:

```
агент строит карту workspace (реальные файлы/директории)
  ↓
агент определяет, где находится нужный модуль
  ↓
агент читает конкретный файл
  ↓
агент принимает решение / запрашивает патч
```

Требование:

- агент должен иметь **актуальную карту проекта** в системном контексте;
- агент должен уметь по инструменту быстро получить **дерево каталогов и файлов** (`describe_workspace`, `list_dir`);
- новые файлы должны определяться как часть реального состояния проекта, а не через догадку LLM.

Иначе получается ложная автономность: инструментов много, но агент не знает, что у него перед глазами.

## Защитный слой (реализация)

1. **Инструмент `request_patch`** (`src/tools/impl/patch_request_tool.py`):
   - Вход: `path`, `user_goal`.
   - Агент (код): читает файл, формирует один запрос к LLM: «по цели пользователя верни только новое содержимое файла».
   - Агент: проверяет ответ (не пусто, не удаление >30%).
   - Агент: вызывает `propose_file_edit(path, new_content)` и возвращает результат пользователю.

2. **Промпт в диалоге:** при запросах на изменение кода использовать **только** `request_patch(path, user_goal)`; не вызывать `write_file` и `propose_file_edit` напрямую для таких запросов.

3. **Guardrails** (уже есть и усилены):
   - запрет удаления, лимит 30% по объёму и по **числу изменённых строк** (diff-based);
   - **жёсткий лимит** `MAX_PATCH_LINES = 200` (патч не больше 200 изменённых строк);
   - **PROTECTED_PATHS**: агент не может менять `src/main.py`, `src/core/`, `src/tools/impl/file_tools.py`, `src/tools/impl/patch_request_tool.py`, `src/governance/`, `src/hitl/` (self-protection);
   - backup перед записью;
   - **syntax gate**: для `.py` перед pytest — `ast.parse(content)`; при ошибке — откат;
   - **test gate**: после `write_file` — pytest; при падении — откат из .bak;
   - в **request_patch** по возможности — **unified diff**, затем валидация.
   - **Sandbox** (`src/evolution/sandbox.py`): при перезаписи файла и включённом test gate запись идёт не в основной проект, а в копию проекта во временной директории; в копии применяется патч и запускается pytest; только при успехе изменённый файл копируется в основной проект. Реальный проект не меняется до прохождения тестов в sandbox.

## Зрелость и оставшийся риск

**Уровни защиты (8):** LLM isolation, unified diff, diff ratio limit, hard patch limit (MAX_PATCH_LINES), syntax gate (ast.parse), **sandbox** (копия → патч → pytest → commit), pytest gate, rollback, protected paths.

**Что агент практически не может:** удалить файл, переписать файл целиком, внести огромный патч, сломать синтаксис, сломать тесты без отката, изменить системные модули (PROTECTED_PATHS).

**Оставшийся риск — семантически плохие патчи:** патч может проходить синтаксис и тесты, но ухудшать логику (например `except: pass`, `return None` вместо реальной обработки). Это общая проблема coding-агентов. Опциональное усиление: semantic diff heuristic — отклонять патчи, где добавлены голый `except:` без обработки, `pass` после except, или удалены определения функций. Пока не реализовано; при необходимости можно добавить как дополнительный фильтр.

**Evolution drift** ограничен **evolution budget** (MAX_EVOLUTION_PATCHES_PER_CYCLE): не более N accept за цикл, чтобы агент не тратил циклы на бесконечные «улучшения» (improve logging, refactor), которые проходят тесты, но не дают пользы. Опционально на будущее: *regression guard* — coverage/performance не хуже после патча.

**Возможные обходы patch_guard (и смягчение):**
- Обход через новые файлы в той же области (*file_a.py*, *file_a_utils.py*, *file_a_helper.py*): при необходимости включить **MAX_PATCHES_PER_DIRECTORY** (env, по умолчанию 0 = выключено) — лимит суммарных патчей на директорию.
- Цепочка «маленьких» патчей уже ограничена **MAX_PATCH_LINES** и diff_ratio (30% строк).

**Координация нескольких агентов (multi-agent):** реализован **evolution lock** (`src/governance/evolution_lock.py`): только один агент в момент времени может выполнять `accept_patch_to_stable`. Остальные получают сообщение «another agent is applying a patch (holder: …). Wait and try again». Lock хранится в `data/evolution_lock.json`, TTL по умолчанию 120 с (env `EVOLUTION_LOCK_TTL_SECONDS`). В Telegram: **/guard** — полный статус (patch_guard, task_guard, evolution lock, семья); в **/status** добавлена строка про очередь и evolution lock с подсказкой «/guard».

## Принцип

**Автономный агент — это в первую очередь инженерия, потом LLM.**

- Очереди задач, циклы, инструменты, метрики, Telegram — это уже большая часть агента.
- Не хватало защитного слоя между агентом и LLM в диалоге — он вводится через `request_patch` и правила использования инструментов.
